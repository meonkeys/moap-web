3. Functional Requirements

These are the high-level functional requirements of the Hisaab software module.

Module A - Client / Internal Accounting (Meeting Leader)

  • 3.1 Group Meeting Accounting

    The software should be able to handle Group Internal Accounting, either in the format of a SHG Meeting, or by entering one transaction at a time. These transactions should be dual-verifiable - when the transaction is recorded it should be clear to both the Group Leader and the Client that the correct information has been recorded. For this reason appropriate visual feedback is very important. These are the following kinds of transactions that must be processed:

    • Collect Subscription Fee / Deposit
    • Collect Loan Principal Repayment
    • Collect Loan Principal Repayment
    • Disburse a Loan
    • Collect a Fee
    • Record Operational Expense

    For each transaction, the following data at least should be recorded:
    • Transaction Type
    • Transaction Date
    • Scheduled Date
    • Amount
    • Debit / Credit
    • Account ID

  • 3.2 Client Account Features (Extra)

    The system should also give the end-client access to information about his/her account, either on-screen or as a printed document. The Client will possibly even less literate and educated than the Group Leaders, so effort should be made to present data in as accessible a manner as possible. Even though they wont have to interact with the system very much (if at all), they should still be able to see and visualize details about their accounts, such as the following:

    • Account Balance
    • Account History
    • Account Statement
    • Period Transaction Statement
    • Payment / Deposit Schedule
    • Trends / Porfolio Analysis

Module B - Group / External Accounting (Group Accountant)

  • 3.3 Group External Account Management

    The system should also be able to display a list of the external accounts (and possibly investments) of the Group itself. This includes savings accounts with banks, loans taken from banks, savings in other institutions, etc. The offerer of these financial services need not be a bank, it could be the Federation or possibly in the future even another Group. This data need not be entered nor reside locally. It could be incrementally downloaded or be available on demand from a central data source, if that facility is possible. For each account, the following operations / transactions should be possible.

    Savings Accounts

    • Open a new account (enter bank / org details, account number, etc.)
    • View / download current liquid balance
    • View / download current total balance
    • View / download transaction history
    • Record / download deposits / withdrawals
    • Record / download interest / dividend income
    • Subtract fees / charges
    Credit Accounts
    • Record / download a Loan (enter bank / org details, loan terms, repayment schedule, etc.)
    • View / download current outstanding
    • View / download transaction history
    • View / download payment schedule
    • Record loan interest / principal repayments (or download)

  • 3.4 Group Accounting

    The system should be able to track the overall financial status of the Group. This would include tracking current and historical accounts, cash in hand, liabilities, income, expenses, etc. It should provide the following: (We need to hash this out better)

    • Daily Cash Book
    • Cash Flow Projections
    • Record Operational Expense / Petty Cash

    The system should also be able to pass on some portion of the income / loss of the group as interest / dividend / fees to the members.
    Sources of income:

    • Member loan interest payments
    • Member fees
    • External Savings Interest / Dividend Accrual
    Sources of Expenses:
    • External Loan Fees / Interest
    • Operational Costs

  • 3.5 End of Period Processing

    The system needs to handle end-of-period processing, including processing all interest/ dividend / fee calculations at the end of each accounting cycle. Can we describe these processes better?

  • 3.6 Reporting (Extra)

    The system must generate a range of reports appropriate for SHGs. It should output these Reports into an electronic format that can be printed, delivered or stored, and is accessible from variety of system platforms. HTML or PDF could be a good choice. Here are a list of reports that could be generated: What reports would it be useful for a SHG to produce?

    • Reports for External Agencies (Bank, FA, etc.)
    • Daily Reports
    • Daily Delinquency Reports
    • Daily Transaction Reports
    • Monthly, Quarterly, Annual Financial Statements
    • Audit Trail
    • Customer Balances, Histories, Statements, Queries
    • Others? Customizable?

Module C - Administration (Group Leader / MFI Rep)

  • 3.7 Group Management

    The system will need to manage group, user and account information. This would include the creation, modification and deletion of Users, Groups and Accounts. In the initial stages this will also be configured by the external staff member upon installation, but eventually we would hope to provide a simpler interface which could allow the Group Leader to handle some of these Group Management tasks. These are the tasks that would need to be processed:

    • Create a Group
    • Modify a Group
    • Close a Group
    • Add a Member to a Group
    • Remove a Member from a Group
    • Create a User
    • Modify a User
    • Delete a User
    • Create an Account for a User
    • Close an Account for a User

    This is the information that would have to be entered on the creation / modification of a Group:

    • Group ID
    • Group Name
    • Date Started
    • Group Address
    • Group Formal Representative (to banks, etc.)
    • Group Leader
    • Group Accountant
    • Other Group Officials
    • Group Members
    • Set Meeting Schedule
    • Set Available Financial Instruments
    • Demographics (caste, race, etc.)

    This is the information that would have to be entered on the creation / modification of a User:

    • Client ID
    • User Name
    • User Address
    • Date Joined
    • Demographics (age, d/o/b, sex, caste, race, etc.)

    This is the information that would have to be entered on the creation / modification of a Account:

    • Client ID
    • Source ID
    • Account ID
    • Financial Instrument Type
    • Date Opened

    The system should also allow the recording of information for external organisations such as Banks, MFIs, Federations, other Groups, Funding Agencies, etc. Here is the info that should be recorded for each:

    • Agency Name
    • Agency Address
    • Agency Type (Bank, NGO, FA, MACS, etc.)

  • 3.8 Security / Administration

    The system would have to maintain a login / password access for all administrative and processing tasks. Even if a separate Computer Operator is actually operating the computer, the password should be entered by the Users themselves and not be told to anyone. There would be several distinct privilege classes, and each administrative user would be assigned priveleges from amongst these classes. The basic sets of users would include - 1) the "root", core administrator, typically an external staff member, 2) four levels of Group Leaders, over time as the group becomes more mature and the Leader more experienced, more priveleges to set Group Structure and Policies would be given to the Group Leader and 3) the end Client. These are the specific privelege classes and the allowed operations for each:

    • Super-User
      • Assign User Priveleges
      • Create / Modify /Delete Groups
    • Group Manager (Level III) - For own group:
      • Remove Members
      • Close Accounts
      • Set / Modify Financial Instruments
    • Group Manager (Level II)
      • Add / Modify Members
      • Open / Modify Accounts
      • Set Group Leader / Accountant
      • Set Available Financial Instruments
    • Group Manager (Level I)
      • Set Meeting Schedule
      • Group External Accounting
      • Data Transfer to Central MIS
    • Group Manager (Level 0)
      • Group Internal Accounting
    • Client
      • View Personal Account Details / History

  • 3.9 Financial Instrument / Business Rules Definition (Extra)

    The system should allow the definition of a number of Financial Instruments (both savings and credit) to be offered within the SHG, specification of the parameters defining these Financial Instruments, and configuration of the Business Rules surrounding the management of these Financial Instruments.

    Initially the Financial Instruments to be offered and the business rules to be followed will be configured by an external staff member during installation and training with the software, possibly after consultation with the Group Leader and Members. This will ensure that a simple, appropriate set of instruments and rules will be implemented by the group in the early stages. In the future we could develop a more accessible interface to this definition process which would allow the Group Leaders themselves to modify and add new financial instruments.

    These are the parameters that can be set for each Savings Financial Instrument:

    • Name - The name of the instrument
    • Interest Rate - The interest that is earned on the savings account
    • Interest Calculation - The way interest is calculated and compunded
    • Dividend Amount - The amount of dividends earned on the Account (could be Variable)
    • Dividend Frequency - The frequency of dividends
    • Fee Amount - Fees charged for opening the Account
    • Fee Frequency - The frequency of Fee payment
    • Withdrawal Restrictions (Current vs. Term) - Set the restrictions on withdrawals from this account - including the following:
      • Max Amount - The max amount that can be withdrawn in one transaction / day
      • Fee - The Fee charged to each withdrawal
      • Frequency - How often withdrawals can be taken (monthly, weekly, yearly, etc.)
      • Term - Duration after opening the account after which a withdrawal can be taken
    • Deposit Requirements (Voluntary vs. Compulsory) - Set the requirements for Deposits to this account: including the following:
      • Maximum Amount - The maximum amount that must be deposited at each Deposit
      • Minimum Amount - The minimum amount that must be deposited at each Deposit
      • Fixed Amount - The fixed amount of each deposit
      • Frequency - The frequency when deposits must be taken

      These are the parameters that can be set for each Credit (Loan) Financial Instrument:

      • Name - The name of the instrument
      • Max Amount - The maximum amount that can be lent under this instrument
      • Min Amount - The minimum amount that can be lent under this instrument
      • Interest Rate - The Interest Rate charged to this Loan
      • Interest Rate Calculation - The way Interest is calculated and compounded
      • Fee Amount - Any fee charged for the loan
      • Fee Frequency - The frequency of the above Fee payment
      • Repayment Frequency - When repayments are expected
      • Term - The term of the loan
      • Purpose - Consumer / Commercial / Emergency
      • Tied to Savings Product - If this loan is tied to an associated Savings Product
      • Late Payment Fee - What is the fee for late Payments
      • Late Payment Scheduling - How are late payments scheduled
      • Early Payments - Are early payments allowed
      • Balloon Payments - Are balloon payments allowed

  • 3.10 Date Exchange with Central MIS (Extra)

    The system should be able to collate all of the data gathered over an accounting / transaction cycle and to upload all of the relevant data to the central MIS system, which could reside at the office of the Apex Institution, the bank or the supporting MFI. The types of information that need to be uploaded include the following:

    • New Members / Accounts
    • Closed Accounts
    • New / Modified Financial Instruments
    • Client Transactions
    • Group Transactions

    Additionally, the system will need to download information at the beginning of each accounting / transaction period. This would include the following information:

    • Account Details / Balances
    • Interest / Dividend Earnings on External Savings Accounts
    • External Payments Due / Scheduled
    • Interest Rate / Financial Instrument Changes

    The operation of this data transfer module should be flexible and secure. The data should be able to be transferrd either via dial-up to the central server, via the internet, or via physical media like a Zip disk or CD. The data will need to be read by a partner program installed on the server-side, which will parse the data and integrate into the local DB securely. This data will have to be authenticated before it can be integrated centrally, possibly through some password or public / private key mechanism. XML is the preferable data transfer format.

Extra Features

  • 3.11 MFI Messaging / Feedback (Extra)

    The System should provide some facility to exchange Messages (through the Data Transfer System) to the supporting MFI and / or other external agencies. These messages could be audio or text-based depending on the literacy of the users. The system should provide a simple, in-context mechanism to send and receive general messages and responses. It is not essential that these messages and responses be targeted to a particular user or person, it should be seen as a general mechanism for SHGs to exchange feedback and ideas with the supporting agency / representative.
